# conkit-sketch Crate Architecture

`conkit-sketch` is the Contract Kit domain library for language-neutral code snippet
contracts. It is storage-agnostic and byte-in, byte-out: callers provide
logical catalog names plus bytes, and the crate returns diagnostics, reports,
and updated combined-contract bytes in catalogs.

Operational rules and validation commands live in [AGENTS.md](AGENTS.md). This
document owns the structural boundary, public surface, data flows, and module
map.

## External Domain Boundary

The dependency direction is one-way:

```text
embedded caller --------> conkit-sketch
storage adapter --------> conkit-sketch

user and operating system
          |
          v
       conkit -----------> conkit-sketch
          |
          +-------------> conkit-signature

conkit-signature !-> conkit-sketch
conkit-sketch    !-> conkit-signature
conkit-signature !-> conkit
conkit-sketch    !-> conkit
```

`conkit` is one caller, not a required gateway. `conkit-sketch` does not know whether
catalog bytes came from local disk, memory, GitHub, object storage, a database,
an embedded application, or a test fixture.

Callers own:

- filesystem discovery and directory walking
- converting operating-system paths into logical catalog names
- reading and writing bytes through any storage provider
- output roots, merge policy, collision policy, and persistence
- command grammar, terminal output, process exits, and user-facing errors
- cross-domain orchestration between signature and sketch contracts
- any signature-assisted discovery that produces snippet seeds

`conkit-sketch` owns:

- logical catalog path validation and deterministic in-memory catalogs
- sketch YAML parsing and contract validation
- versioned language-neutral exact-line normalization
- matching, diagnostics, check counts, and check-mode behavior
- semantic sketch catalog diffing
- YAML and JSON report bytes
- linked-sketch refresh inside combined contract documents
- runtime-neutral scheduling of its CPU-bound work

The crate builds only the minimal signature index needed to resolve top-level
signature-owned sketch links: document, label, file, and signature type. A
caller may use signature-derived source spans to construct `SketchSeed` values,
but `conkit-sketch` does not parse Rust or depend on `conkit-signature`.

## Public Surface

`SketchContractKit` is the public behavior handle and is constructed through
`SketchContractKitBuilder`.

Public async operations:

- `check(CheckRequest) -> CheckResponse`
- `generate(GenerateRequest) -> GenerateResponse`
- `diff(DiffRequest) -> DiffResponse`

Public boundary types:

- `FileCatalog`, `CatalogPath`, and `FileCatalogError`
- `SketchContractKitError`
- `CheckRequest`, `CheckResponse`, and `CheckMode`
- `SketchMatchPolicy`, `SketchNormalization`, and `SketchOccurrence`
- `SketchCheckCounts`, `SketchDiagnostic`, `SketchLocation`,
  `SourceLineSpan`, `MatchCandidate`, and `DiagnosticExcerpt`
- `ReportRequest` and `ReportFormat`
- `GenerateRequest`, `GenerateResponse`, and `SketchSeed`
- `DiffRequest`, `DiffResponse`, and `DiffEntry`
- `WorkOptions`, `WorkerPool`, and the public sketch limit types

Public DTOs use in-memory catalog bytes and logical catalog paths. They do not
contain `Path`, `PathBuf`, filesystem roots, storage-provider handles, or lists
of locally written files. Output-producing operations return `FileCatalog`
values so callers choose where and how to persist them. Requests, responses,
and their public enum fields remain serializable, deserializable, and
comparable at this boundary.
`CheckResponse::report_view` exposes an opaque borrowed serializable view so
standalone and cross-domain callers reuse the domain-owned report schema
without another report DTO or diagnostic clone.

The builder configures scheduling and resource budgets independently.
`WorkerPool` selects a runtime-default pool, a dedicated pool with an explicit
worker count, or an application-owned shared `Arc<rayon::ThreadPool>`.
`WorkOptions::max_in_flight_operations` bounds active root work, while
`max_pending_operations` bounds admitted waiters. `SketchLimits` separately
bounds catalogs, YAML parsing, identifiers, snippets, normalized sources,
diagnostics/evidence, and generated bytes.

## Catalog Boundary

`FileCatalog` is a private-map wrapper over deterministic logical path order.
It rejects duplicate paths rather than replacing existing bytes and exposes
only catalog-oriented accessors and iteration.

`CatalogPath` is a UTF-8 logical name, not an operating-system path. It is:

- nonempty and relative
- separated with `/`
- free of backslashes, colons, and NUL bytes
- free of empty, `.`, and `..` components

It serializes as a scalar string and deserializes through `CatalogPath::new`.
That allows a nonempty catalog to round-trip through JSON map keys without
bypassing path validation.

The byte boundary is intentionally broader than UTF-8 source text. The
`exact_lines_v1` kernel operates on arbitrary bytes without selecting a
different algorithm for malformed UTF-8. Resolved generation seeds carry UTF-8
strings because they replace YAML scalar content.

## Check Data Flow

```text
CheckRequest
  source FileCatalog
  contract FileCatalog
  mode
  report request
        |
        v
SketchContractDocuments::from_catalog
  -> parse each direct root-level .yaml/.yml document once
  -> retain original selected-document bytes
  -> retain nested and non-YAML entries as passthrough
  -> contracts() validates links and builds SketchContracts
        |
        v
SourceCatalog::from_catalog
  -> retain raw bytes only for referenced source entries
        |
        v
SketchMatcher
  -> group sketches by source path and normalization policy
  -> normalize each source group once
  -> evaluate AtLeastOne or ExactlyOne within each source group
  -> compute bounded nearest-candidate evidence only after a miss
  -> deterministic diagnostic sort
        |
        v
SketchInventoryComparison
  -> counts and mode-dependent passed value
        |
        v
ReportRequest::render
  -> optional YAML/JSON report FileCatalog through a borrowed domain view
        |
        v
CheckResponse
```

`source_catalog_entry_count` describes the complete supplied source catalog.
`referenced_source_file_count` and `present_referenced_source_file_count`
describe unique logical paths selected by sketches and the subset present in
that catalog. `contract_document_count` counts semantic YAML documents rather
than physical catalog entries, including valid documents with no linked
sketches.

Report generation happens after comparison and serializes only `passed`,
`counts`, and `diagnostics`; it does not recursively serialize `report_files`.

## Contract Format And Parsing

Sketches live inside the canonical combined document shape:

```yaml
contract_version: 2
root: ../src
files: [lib.rs]
extraction:
  mode: rust_syntax_v2
  profile: rust_api_v1
  crates:
    - id: example
      root: lib.rs
      kind: library
signatures:
  - answer_signature:
      file: lib.rs
      signature_type: function
      name: answer
      sketch: answer_body
sketches:
  - answer_body:
      file: lib.rs
      signature: answer_signature
      signature_type: function
      matching:
        normalization: exact_lines_v1
        occurrence: at_least_one
      code: |
        pub fn answer() -> u8 {
            42
        }
```

The parser considers direct root-level case-insensitive `.yaml` and `.yml`
catalog entries and ignores nested YAML. Each selected document must declare
`contract_version: 2`, `root`, `files`, `signatures`, and `sketches`; use
`sketches: []` when the list is empty. Signature-owned `extraction` data is
preserved without teaching this crate Rust extraction semantics. Missing or
unsupported versions, the `version`/`language` shape, duplicate keys, and
other mixed shapes are rejected. The `conkit-sketch` crate validates the
combined facts needed by its domain:

- `root` is a nonempty user-facing string;
- `files` contains unique validated logical paths within that document;
  separate documents may intentionally overlap their allowlists to reuse the
  same source file through document-local projections;
- top-level signature entries have document-local unique nonempty labels, a
  listed `file`, a nonempty `signature_type`, and an optional string `sketch`
  link;
- sketch entries contain exactly one ID whose nested body has `file`,
  `signature`, `signature_type`, `matching`, and `code`;
- sketch IDs are globally unique, preserve their exact scalar spelling, reject
  surrounding whitespace and control characters, and stay within the configured
  byte limit;
- every sketch declares the closed matching policy
  `normalization: exact_lines_v1` plus `occurrence: at_least_one | exactly_one`,
  and code remains nonempty after normalization;
- every sketch is referenced by exactly one top-level signature in the same
  document, every link resolves, and the nested file, signature label, and
  signature type agree with that link.

The `conkit-signature` crate owns complete validation of all other signature
fields; `conkit-sketch` intentionally does not duplicate a Rust signature
parser. Malformed YAML, unknown root fields, duplicate document-local labels or
global sketch IDs, invalid paths, orphan, ambiguous, missing, or cross-document
links, kind mismatches, and empty values fail with `SketchContractKitError`
before matching. A missing target source file, a valid snippet that does not
match, or an occurrence-policy violation produces a check diagnostic instead.

Each root operation owns one cumulative catalog counter and one cumulative
YAML budget. One cancellable raw event pass simultaneously discovers physical
document metadata and charges documents, depth, nodes, aliases, and scalar/tag
bytes transactionally; typed semantic parsing then charges only replay beyond
that raw report. Check shares catalog entry/byte limits across source and contract
inputs; diff shares them across both contract sides; generation accounts its
complete contract input. The same YAML budget spans every selected physical
contract file, both sides of a diff, and every changed-document semantic
verification reparse, so per-file parsing cannot reset an operation budget.
Raw nodes and scalar bytes are charged once during resource preflight; the
final typed-parser report charges only alias-replayed nodes and scalar bytes
beyond that raw report. Semantic parsing receives only the operation's
remaining replay capacity while still being allowed to consume the current
file's already-recorded raw events. Its internal replay-event cap is twice the
remaining node count because a balanced container contributes one semantic
node but both a start and an end event; the semantic node budget remains the
exact public authority rather than exposing a second replay-event limit. A
replay-cap diagnostic converts events back to the proven lower bound of one
node per two events before composing operation-wide evidence.

## Normalization And Matching

Contract snippets and selected source files pass through the same closed,
versioned language-neutral normalization kernel. `exact_lines_v1`:

- converts CRLF line endings to LF;
- treats one final LF or CRLF terminator as nonsemantic;
- preserves additional final blank lines;
- preserves leading, internal, and trailing horizontal bytes;
- preserves indentation, tabs, and internal blank lines;
- preserves isolated carriage returns and arbitrary invalid UTF-8 bytes; and
- never changes its policy based on whether a complete line decodes as UTF-8.

This conservative policy deliberately does not collapse whitespace or drop
blank lines. String literals, comments, indentation-sensitive languages,
Makefile recipes, raw/multiline strings, heredocs, and embedded data therefore
retain their byte-level meaning.

`SketchSnippet` performs this normalization once when a contract is
constructed. Matching and semantic comparison borrow the cached
`NormalizedSnippet` instead of rebuilding it.

Semantic diffing keys entries by sketch ID. For the same ID, comparison uses
the linked logical source file, linked signature label, `signature_type`,
explicit matching policy, and normalized code. The containing contract
document, YAML formatting, YAML comments outside `code`, and mapping order are
nonsemantic. Comments and other bytes inside the `code` scalar pass through
exact-line normalization with the rest of the code, so they remain semantic.
Added and removed entries carry the snapshot from the side where the sketch
exists; changed entries carry previous/current snapshots plus deterministically
ordered typed field flags. Snapshot contract path and document index are
locator evidence only. Code and complete-current-contract SHA-256 digests use
version 2, explicit domain separators, stable policy tags, and length-framed
semantic bytes; `changed()` is derived from whether diff entries exist.

A sketch checks only the logical source path named by its linked signature.
Sketches are grouped by `(source path, normalization)` so a referenced source is
normalized once per policy and forms the coarse scheduling unit. Matching stays
on the grouped linear baseline until Phase 9 benchmarks justify an index.
One operation-wide matching ledger bounds exact normalized-line comparisons and
encountered occurrence candidates across every source group. Crossing either
budget returns typed `LimitExceeded` evidence for the participating source
instead of truncating search, resetting work per sketch, or returning a false
match result. Retained-span and excerpt limits remain presentation budgets;
they do not weaken the exact occurrence count or comparison semantics.

`AtLeastOne` accepts the first contiguous ordered occurrence. `ExactlyOne`
counts overlapping occurrences, retains the exact count, and stores only the
configured number of one-based inclusive spans in ascending source order. Zero
occurrences produce `NotMatched`; duplicate unique-policy occurrences produce
`OccurrenceMismatch` with an exact count and an explicit span-truncation flag.

The fast success path does not build mismatch evidence. After a miss, a bounded
diagnostic pass scores aligned windows by equal lines, chooses the earliest
source line on ties, and reports the first differing expected/source line.
Arbitrary bytes are escaped safely, and a missing aligned source line is
represented explicitly. Missing-file, mismatch, and occurrence diagnostics all
carry the sketch ID, contract path, zero-based YAML document index, and source
path.
Candidate search retains only source coordinates. Before allocating escaped
excerpts, diagnostic construction reserves the next diagnostic count, measures
the exact serialized skeleton plus ASCII and JSON escaping in a cancellable
pass, and rejects the first aggregate-byte crossing without committing meter
state. Only an accepted reservation is materialized and recorded, so limit or
cancellation failure retains neither evidence text nor partial accounting.
Diagnostics are sorted afterward by complete location and diagnostic kind so
worker scheduling cannot change returned or rendered results.

Mode behavior is:

- `Enforce`: fail when diagnostics exist
- `Warning`: retain diagnostics and set `passed` to `true`

Each parsed sketch contributes at most one diagnostic. Therefore
`failed_sketch_count` equals `diagnostics.len()`, and `matched_sketch_count`
equals `sketch_count - failed_sketch_count`.

## Generation Data Flow

```text
GenerateRequest
  combined contract FileCatalog
  Vec<SketchSeed>
  GenerateMode::{FullRefresh, PartialRefresh}
        |
        v
SketchContractDocuments::from_catalog
  -> one parsed combined-document set
  -> contracts() borrows it for global validation
        |
        v
SketchRefreshSeeds
  -> FullRefresh requires one seed per linked sketch
  -> PartialRefresh targets only supplied IDs
  -> validate document, ID, kind, file, and code
        |
        v
SketchContractDocuments::refresh
  -> render only targeted documents
  -> reuse original bytes for untargeted root documents
  -> append nested and non-YAML passthrough unchanged
        |
        v
GenerateResponse { contract_files, SketchGenerationCounts }
```

`SketchContractDocuments` is the sole parsed combined-document owner used by
validation and generation, so a refresh does not parse the YAML catalog a
second time.

Each `SketchSeed` carries `contract_file`, zero-based physical document index,
`sketch_id`, `signature_type`, `file`, and exact `code`. This mirrors the
runtime-neutral output of a caller's signature resolver without coupling the
crates. Seed identities must be unique and every seed must match an existing
linked sketch exactly. Full refresh additionally requires every linked sketch
to have one seed. Partial refresh preserves unspecified sketches, and an empty
partial request is an exact byte no-op. The document locator disambiguates
refresh targets but is not part of semantic diff identity. Code must remain
nonempty after normalization.

Generation reports linked and refreshed sketches separately. A targeted exact
scalar equality is refreshed but unchanged; `changed_sketch_count` uses exact
scalar equality rather than normalized equality, and
`changed_document_count` counts distinct physical YAML documents containing
real edits.

Generation is a combined-document update, not sketch discovery. It changes
only each targeted nested sketch body's `code` scalar. Root values, file allowlists,
signatures, signature-owned links, IDs, and kinds remain intact. Documents
without links, nested YAML entries, and non-YAML catalog entries remain
byte-for-byte unchanged. A
fresh combined document with no explicit sketch links therefore produces zero
sketches rather than creating whole-file sketches.

The semantic parser is authoritative. Exact no-ops return the original bytes
without constructing a lossless tree. Real edits navigate the lazy lossless
syntax tree to the nested scalar. One scalar editor owns style inspection,
preferred encoding, deterministic safe fallback, tags, line endings,
indentation, CST validation, and exact semantic-string validation. A verified
edit set proves replacement ranges are in-bounds UTF-8 boundaries, sorted,
disjoint, and limited to independently validated scalar candidates while
streaming untouched source bytes exactly. The complete edited document is then
reparsed semantically and returned only when it equals the proposal; that
verification consumes the same YAML budget as the initial parse. Unsupported
alias/anchor mutations fail before output is produced.

Returned bytes and simultaneously retained generated/editing text use
independent nominal budgets. Target discovery retains only validated ranges,
lightweight syntax handles, presentation metadata, and borrowed seeds. The
editor renders one replacement through a release-on-drop scratch owner, streams
it into the returned-output writer, and releases it before the next target.
Signature-owned `extraction` and `signatures` entries are compared afterward as
exact borrowed source slices, including the split compatibility shape, without
copying or hashing preservation evidence. Returned output and live scratch each
default to 512 MiB and may be configured independently.

Line-ending selection is local to each target: the scalar's first newline wins,
then its containing physical document, then LF. LF, CRLF, and CR are preserved
as distinct presentation styles; conversion changes only logical line breaks,
keeps isolated CR content bytes, and reconciles a generated trailing break with
the untouched source suffix without producing a hybrid or doubled boundary.

## Runtime And Dispatch

The public handle is an opaque struct that directly owns its work pool and
resource limits. Each public receiver method calls `AsyncWorkPool::execute`
exactly once per operation around the complete receiver-owned
`CheckRequest::run`, `GenerateRequest::run`, or `DiffRequest::run` workflow.
The pool first attempts active-plus-pending admission and returns queue-full
immediately when that capacity is exhausted. An admitted waiter then acquires
an active-operation permit asynchronously before submitting to Rayon. Both
permits move into the worker closure and are released before completion is
sent through the runtime-neutral oneshot channel. The closure best-effort skips
work when its receiver was canceled, catches a job panic for forwarding, and
passes a cooperative cancellation probe through domain checkpoints. A panic
resumes on the polling thread; recoverable failures continue through the typed
error boundary.

Calling an operation directly with `.await` is the normal integration. A
spawned task instead owns its request and kit, commonly through an `Arc` clone
captured by `async move`; that owning future and its output satisfy
`Send + 'static`. A future directly borrowing a local kit is not itself
promised to be `'static`.

Dropping an admitted waiter releases its admission permit. Dropping a submitted
future signals cancellation; parsing, source groups, matching/diagnostic
batches, and rendered documents stop at their next checkpoint. Cancellation is
cooperative rather than unsafe thread termination. Neither semaphore admission
nor Rayon scheduling promises FIFO order. Operations have no external side
effects, so a discarded result cannot leave partial filesystem or network
state.

Within a check job, the admitted root workflow runs on the configured Rayon
pool while source groups and sketches are evaluated sequentially. It does not
fan out one task per sketch. Deterministic sorting occurs before values cross
the public boundary.

## Module Ownership

- `lib.rs`: crate policy, module declarations, and intentional public
  re-exports.
- `api.rs`: public requests/responses, builder, opaque handle, direct
  top-level async work submission, and request-owned complete check and
  current-then-previous diff workflows.
- `files.rs`: `FileCatalog`, `CatalogPath`, deterministic catalog access, path
  validation, duplicate rejection, and catalog errors.
- `error.rs`: the public builder/check/generate/diff error wrapper and
  contextual internal error construction.
- `id.rs`: the private exact sketch-ID invariant shared by declarations, links,
  and seeds; it rejects invalid spelling without trimming or Unicode
  normalization.
- `work.rs`: `WorkOptions`, `WorkerPool`, independent admitted/running
  capacities, cooperative cancellation, panic forwarding, and runtime-neutral
  oneshot completion.
- `limits.rs`: nominal sketch-domain public limit facade, typed
  `LimitExceeded` evidence, and the private charge/conversion mechanics shared
  by its children.
- `limits/catalog.rs`: cumulative logical catalog admission.
- `limits/yaml.rs`: transactional one-pass raw YAML metering and semantic
  replay accounting.
- `limits/matching.rs`: sketch identity, normalization, occurrence-comparison,
  and retained-evidence ceilings.
- `limits/output.rs`: transactional diagnostic reservations plus independent
  scratch and returned-output accounting with first-failure preservation.
- `contract.rs`: parsed-contract facade.
- `contract/model.rs`: canonical sorted sketch collection, validated model,
  locators, and per-contract refresh-seed agreement.
- `contract/document.rs`: strict combined-document parsing, original-byte and
  passthrough preservation, and one-pass event analysis.
- `contract/resolve.rs`: minimal signature-link index, catalog-wide agreement,
  and resolved contract construction.
- `contract/diff.rs`: ordered two-way semantic comparison without rebuilt
  lookup trees, plus versioned code and contract digests.
- `contract/edit.rs`: targeted verified edit-set construction and final
  semantic reparse.
- `contract/edit/scalar.rs`: cohesive scalar presentation, preferred/fallback
  codec, validation, and scratch ownership.
- `normalize.rs`: the contiguous byte buffer and line ranges for
  `exact_lines_v1`, including CRLF handling and exact line comparison.
- `matcher.rs`: referenced-source selection, source-policy grouping, grouped
  linear occurrence evaluation, and bounded nearest-candidate evidence.
- `inventory.rs`: contextual diagnostics, escaped evidence, source spans,
  deterministic diagnostic ordering, counts, comparison invariants, and
  check-mode response construction.
- `report.rs`: YAML and JSON report rendering into a returned catalog.
- `generate.rs`: `GenerateRequest::run`, seed-set admission, uniqueness and
  coverage validation, generated-code validation, and generation
  orchestration; per-contract seed agreement remains model-owned.
- `tests/public_api.rs`: the single integration-crate root over
  `public_api/{support,async_contract,boundaries,workflows}.rs`.

No module in this crate owns filesystem IO, CLI presentation, cross-domain
reports, Rust parsing, or source-span resolution.

## Determinism And Failure Boundaries

Determinism is part of the crate contract:

- catalogs use ordered maps and reject duplicate logical paths
- contract IDs are unique across the complete parsed catalog
- source-group diagnostics are sorted by complete location before response
  construction
- refreshed output files retain deterministic catalog path order
- unaffected documents and non-YAML entries retain their original bytes
- report bytes and refreshed YAML derive from validated ordered values

Recoverable catalog, limit, parse, conversion, render, inventory, queue, and
worker failures return through the public typed error. Overall correctness
diagnostic budgets fail the operation rather than truncating results; only
explicit excerpt/span evidence uses bounded presentation with a truncation
flag. Match failures remain data in `SketchDiagnostic` and affect `passed`
according to `CheckMode`.

## Tests And Workspace Boundary Policy

Unit tests live beside private implementation details. The local `work.rs`
tests deterministically cover worker execution, bounded admission, cancellation
before admission, best-effort queued cancellation, started-job completion,
maximum active root work, panic forwarding, and worker-channel loss. Their
coordination uses channels, atomics, explicit manual polling, and bounded wake
guards rather than sleeps or production test seams.

Public API behavior, serialization, and async-contract tests live under
`conkit-sketch/tests`.
`tests/public_api.rs` remains one integration-crate root over focused support,
async-contract, boundary, and workflow children. The async child asserts at
compile time that directly borrowed operation futures are `Send`, that the
public builder and kit satisfy their ownership contracts, and that `Arc`-owned
spawned-operation futures and outputs satisfy `Send + 'static`. Representative
executor-neutral workflows continue to poll operations through the lightweight
test executor.

Together, unit and public API tests exercise exact whitespace/blank-line
semantics, CRLF/LF equivalence, arbitrary invalid bytes, both occurrence
policies, overlapping matches, contextual candidate/span evidence, exact ID
validation, catalog serde, async checking, both report formats, linked-sketch
refresh, check-mode behavior, binary catalog entries, refreshed-contract round
trips, semantic added/removed/changed diff entries, and no-op diffs for
formatting, YAML comments, mapping order, and contract-file relocation. Token,
comment, indentation, and whitespace changes inside sketch code remain
semantic. The centralized `syn`-based AST policy in
`conkit/tests/dependency_policy.rs` prevents workspace boundary regressions
such as filesystem IO, OS-path DTOs, `clap`, Tokio, `async_trait`,
`conkit_signature::` imports, process exits, and production-scope `#[cfg(test)]`
shims. The public API children do not duplicate source scanning or private
source-spelling checks.
